# 文档治理规范

## 1. 基本原则

文档是项目状态和规则的一部分，不是一次性记录。Agent 必须主动维护文档的一致性、时效性和可执行性，不能只修改代码或完成操作而留下过期说明。

- 代码、配置、基础设施、流程和决策变化时，同步检查并更新相关文档。
- 文档描述必须与仓库现状、服务器现状和已确认决策一致。
- 不等待用户逐项指出过期内容；Agent 应主动搜索受影响文档。
- 不保留已失效的“当前分支”“本次任务”“稍后处理”等临时上下文作为长期规则。
- 事实、规则、计划、建议、历史记录必须明确区分。

## 2. 文档类型

### 规范文档

描述长期有效的原则和约束，例如 `.agents/gitflow.md`、`.agents/monorepo.md`。

- 应使用状态无关、可跨分支复用的措辞。
- 不绑定临时分支名、会话、任务或执行者。
- 流程阶段使用准入条件描述，不使用“当前正在某分支，所以……”描述。

### 状态文档

描述可变化的环境或实施状态，例如服务器、部署和迁移状态。

- 必须注明状态适用对象。
- 需要时注明“最后核验日期”，而不是把历史检查结果写成永久事实。
- 状态改变后必须更新、归档或删除旧描述。

### 决策文档

记录已确认的架构决策及理由。

- 区分 `proposed`、`accepted`、`superseded`、`deprecated`。
- 新决策取代旧决策时，必须标记替代关系，不能保留两套相互冲突的现行规则。

### 临时计划

具体任务的开发计划、调研笔记、迁移清单和执行记录统一放在 `.agents/plans/`，详细约定见 `.agents/plans/README.md`。

- 临时计划可以绑定具体分支、日期和任务，因为它不属于长期规则。
- 文件名必须表达主题，例如 `.agents/plans/monorepo-migration.md`。
- 创建时注明状态、目标、范围、完成条件和关联分支。
- 执行过程中维护实际进度、决策和阻塞项。
- 完成或取消后，不得继续将计划当作现行规则。
- 长期有效的结论必须同步到 `.agents/*.md`、ADR、`docs/` 或其他正式文档。
- 没有审计价值的已完成计划应删除；需要保留的内容应精简并迁移到正式历史文档。
- `.agents/plans/` 中的具体任务计划默认被 Git 忽略；需要提交和共享的计划不得只存放于此。

### 历史记录

仅用于审计、迁移记录或事故复盘。

- 必须明确标为历史，不得与当前操作指引混写。
- 历史命令和旧路径不得让 Agent 误认为仍可执行。

## 3. 禁止易过期写法

长期规则中禁止：

- 写死临时分支名，例如“在 `docs/agent-workflow` 上只能……”。
- 使用脱离条件的“当前”“现在”“稍后”“本次”“后续”等相对时间词。
- 把规划中的目录、工具或服务写成已经存在。
- 把一次检查得到的资源信息当作永久事实。
- 已完成迁移后仍保留“待迁移”“当前仅规划”等旧阶段说明。
- 已停止或删除资源后仍在操作指南中要求访问该资源。
- 复制同一事实到多个文档且没有明确事实来源。

应改为条件式和状态式表达，例如：

- 不写：“当前 Agent Flow 分支禁止创建框架。”
- 改写：“在架构方案未获批准、且尚未创建独立实施分支时，禁止创建框架。”
- 不写：“服务器当前运行容器 A。”
- 改写：“最后核验于 YYYY-MM-DD；执行操作前重新查询实际状态。”

## 4. 事实来源与冲突处理

- 仓库结构和代码行为：以当前工作树及可复现测试为准。
- Git 分支和提交状态：以 `git status`、`git branch`、`git log` 为准。
- SSH 参数：在本机操作时以 `~/.ssh/config` 为准，不得复制到公开文档。
- 服务器和容器运行状态：以实时只读检查为准。
- 用户明确确认的架构决策：以最新决策和规则文档为准。
- 私有背景、服务器资产及部署信息：从可选 `.agents/local.md` 及其索引读取；公开仓库不得依赖其存在。

发现冲突时：

1. 先验证实际状态。
2. 判断哪个来源具有更高时效性和权威性。
3. 更新或标记过期文档。
4. 如果涉及未确认的架构选择或破坏性变更，向用户确认。
5. 不得为了保持文档一致而擅自修改实际系统。

## 5. 变更影响检查

完成以下变更时，Agent 必须主动搜索并检查相关文档：

- 分支、Git 流程或提交策略变化。
- 目录、服务、包或模块的创建、迁移、重命名或删除。
- 根或模块级 `AGENTS.md` 的索引、作用域和路径变化。
- workspace、包管理器、构建系统或框架选型落地。
- 服务边界、依赖方向、接口、返回契约、消费者功能或数据所有权变化。
- 修复非显然故障、跨模块兼容问题或可能重复发生的踩坑。
- Docker project、镜像、容器、网络、卷或端口变化。
- SSH 用户、服务器用途、部署路径或权限变化。
- 数据库、环境变量、密钥管理或部署流程变化。
- 计划阶段进入实施阶段，或实施完成进入维护阶段。

推荐搜索：

```bash
rg -n '<旧名称>|<旧路径>|<旧分支>|<旧端口>|<旧状态>' AGENTS.md .agents docs README.md
```

不能只更新直接提到的一个文件；必须检查索引、交叉引用和重复描述。

## 6. 生命周期维护

### 任务开始前

- 读取 `AGENTS.md` 和与任务相关的规则。
- 检查 `.agents/plans/` 中是否存在同一任务的有效计划，避免创建重复计划。
- 验证文档中的路径、分支和环境是否仍存在。
- 如果文档明显过期，先修正文档或向用户说明冲突。

### 任务进行中

- 记录哪些规则或状态将因本次改动失效。
- 修改公开能力时，记录返回契约和具体消费者功能的影响。
- 遇到非显然错误时记录症状、根因、正确做法、禁止做法和验证方式；确认后沉淀到最近作用域的模块 `AGENTS.md`。
- 新增文档时明确其类型、适用范围和事实来源。
- 避免创建内容重叠、名称模糊的新文档。

### 提交前

- 根据 staged diff 搜索相关文档中的旧名称、旧路径和旧状态。
- 如果公开能力或返回结构变化，验证并更新提供方与全部直接消费者的依赖记录和测试要求。
- 检查已解决的踩坑是否需要新增或更新 `DO NOT` / 历史教训；同时删除不再适用的警告。
- 检查根及受影响模块的 `AGENTS.md` 索引是否完整且路径可用。
- 新建模块必须包含模块级 `AGENTS.md`；移动、重命名或删除模块时同步维护所有相关索引。
- 检查文档是否错误引用当前工作分支或临时任务状态。
- 检查规划内容是否被误写成已实施，或已实施内容是否仍被写成待办。
- 文档应与该提交合并后的仓库状态一致，而不是只与提交前工作区一致。

### 合并、部署或迁移后

- 更新实施状态、部署状态和最后核验信息。
- 删除或改写只适用于实施前的限制。
- 完成或关闭对应 `.agents/plans/` 临时计划，并把长期结论同步到正式文档。
- 将仍有审计价值的旧内容移入明确的历史章节；没有价值的直接删除。
- 再次检查所有交叉引用和事实来源。

## 7. Agent 自主治理边界

Agent 可以无需额外确认：

- 修复错别字、失效链接和明确过期的分支名。
- 将临时措辞改成不依赖时间的条件式规则。
- 根据已经完成且可验证的操作更新状态说明。
- 删除与新规则完全重复或已被明确取代的旧说明。
- 更新索引和交叉引用。

Agent 必须先确认：

- 改变架构、服务边界、数据所有权或安全策略。
- 把“建议”升级为“已决定”。
- 删除仍可能代表用户意图的未完成计划。
- 在信息冲突且无法验证时选择其中一个作为事实。

文档治理允许维护描述，不授权 Agent 擅自改变项目或基础设施来迎合文档。

## 8. 完成标准

涉及文档影响的任务只有同时满足以下条件才算完成：

- 相关事实和规则已更新。
- 不存在已知的旧名称、旧路径、旧阶段或冲突说明。
- `AGENTS.md` 索引准确；公共索引指向已跟踪文件，本地私有索引明确标为可选。
- 公开文档不包含服务器地址、私有 SSH 用户、内部部署路径、实时拓扑或其他本地敏感信息。
- 文档明确区分已实施、已决定、建议、待确认和历史。
- 状态信息可通过命令或权威来源重新验证。
- 最终回复说明更新了哪些文档，以及是否仍有待确认事项。
