# TokenMP v3 Monorepo 规范

## 1. 实施准入条件

TokenMP v3 确定采用 Monorepo 管理代码。Monorepo 搭建和迁移必须满足以下全部准入条件：

- 用户已明确确认架构方案和实施范围。
- 实施前必须确认的事项已完成，或未决项已被用户明确接受。
- 已从最新 `main` 创建专用实施分支。
- 工作区不存在来源不明或可能被覆盖的改动。
- 已定义分阶段迁移、验证和回滚方案。

在准入条件未满足时，只允许：

- 调研现有代码、历史实现和技术约束。
- 定义目标目录、服务边界、依赖方向和迁移方案。
- 编写或调整 `AGENTS.md`、`.agents/*.md` 及同类流程文档。
- 提出需要用户确认的技术选型和 ADR 草案。

在准入条件未满足时，禁止：

- 创建 Monorepo 目录骨架。
- 创建或修改 workspace 配置。
- 初始化 pnpm、npm、Yarn、Bun、Turborepo、Nx 等工具。
- 创建 `package.json`、lockfile、`go.work`、`turbo.json`、`nx.json` 等框架文件。
- 移动、复制、重命名或删除业务代码。
- 批量修改 import、模块名、构建脚本、Dockerfile 或 CI。
- 安装依赖、生成模板或运行脚手架。
- 实现业务功能或执行项目结构迁移。

准入条件满足后，应更新本文件中已落地的工具选型、目录和迁移状态，不得继续保留实施前措辞。

## 2. Monorepo 目标

Monorepo 用于统一管理 TokenMP v3 的服务、应用、共享代码、契约、基础设施和文档，同时保持服务可独立开发、测试、构建和部署。

目标包括：

- 一个 Git 仓库和统一的开发入口。
- 清晰表达服务所有权、数据边界及依赖方向。
- 共享真正稳定且跨服务一致的代码和协议，避免复制漂移。
- 每个可部署单元能够独立构建镜像、配置环境和发布。
- 支持按受影响范围执行 lint、test、build 和 deploy。
- 统一版本、质量门禁和开发规范，但不把微服务退化成模块化单体。

Monorepo 是代码组织和工程治理方式，不表示所有服务必须：

- 使用同一种语言或框架；
- 运行在同一个进程；
- 共用一个数据库；
- 一起构建或一起发布；
- 通过源码内部调用替代网络契约。

## 3. 既定架构边界

以下边界属于当前已知约束，后续设计不得擅自改变：

- `auth`：负责认证和 JWT，只访问认证数据库。
- `api`：负责 API Key、业务配置、模型与上游配置、业务及请求日志相关能力。
- `quota`：负责配额检查、预留、结算、释放及流水。
- `executor`：核心转发执行服务，保持 Go 实现，不直接访问数据库；通过可配置的内部 HTTP 接口访问 API 和 Quota。
- Web、Admin 或 Gateway 属于独立应用/可部署单元，不得把核心执行逻辑复制进前端或网关。
- 服务间通过明确、可版本化的契约通信，不得跨服务直接读取对方私有数据库或导入对方内部实现。

如现有代码、历史实现和本规则不一致，应先记录差异并向用户确认，不得静默改变边界。

## 4. 目标逻辑分区

仓库采用以下目标逻辑类别，但只在模块实施范围确认后渐进创建，不提前建立空目录：

```text
apps/        # 面向用户或运营人员的应用、Web、Admin、Gateway
services/    # 可独立部署的后端服务，如 auth、api、quota、executor
packages/    # 经审查后允许跨项目复用的库、契约和工具
infra/       # Docker、Compose、部署、代理、可观测性等基础设施定义
docs/        # 架构、ADR、接口、数据与运维文档
tools/       # 仓库级脚本和工程工具
```

仓库级 workspace、工具链及顶层逻辑分区已经创建。`packages/ui-tokens` 是首个 Node.js workspace 模块；`packages/contracts` 是语言中立 API 契约 package；`services/auth` 是首个 Go 服务（已实现 Auth Identity Flows）。apps、infra 和 tools 当前没有具体模块。后续每个模块必须在实施前由用户确认，并通过独立变更引入。

## 5. 可部署单元规则

每个 `apps/*` 或 `services/*` 可部署单元应保持自治：

- 有明确的负责人、用途和 README。
- 有独立入口、构建命令、测试命令和健康检查。
- 有独立 Dockerfile；构建上下文由仓库根目录统一约定。
- 显式声明运行时配置和环境变量，不依赖隐式本机状态。
- 不通过相对路径读取其他服务的私有源码或运行时文件。
- 可以独立构建和发布，不要求无关服务同时发布。
- Docker 镜像遵循 `.agents/docker.md` 中的 `tokenmp-v3-<service>` 命名规则。

## 6. 分层 AGENTS.md 规范

Monorepo 根目录以及每个可独立开发、测试、构建或维护的模块都必须包含 `AGENTS.md`，使 Agent 能从当前目录快速定位适用规则和文档。

必须包含 `AGENTS.md` 的范围：

- Monorepo 根目录。
- 每个 `apps/<name>/` 应用。
- 每个 `services/<name>/` 服务。
- 每个 `packages/<name>/` 共享包。
- `infra/`、`tools/` 等存在独立操作边界、风险或工作流程的模块。
- 更深层且具有独立技术栈、数据所有权或高风险操作的子模块。

规则层级：

- 根 `AGENTS.md` 是全仓库入口，只索引全局背景、治理规范和跨模块规则。
- 模块 `AGENTS.md` 只描述或索引该模块特有的职责、结构、命令、接口、数据边界、测试和部署规则。
- 模块规则继承根规则，不得重复复制整份全局规则。
- 更深层的 `AGENTS.md` 可以补充更具体的规则，但不得静默放宽上层安全、Git、数据和部署约束。
- 不同层级发生冲突时，应先遵循安全性更高、作用域更具体的规则；如果仍无法判断，停止修改并向用户确认。

创建模块文件时使用 `.agents/templates/module-AGENTS.md` 作为基线，并删除不适用的占位内容，不得原样保留空模板。

每个模块 `AGENTS.md` 至少应索引：

```md
# <模块名>

## 模块职责

## 必读文档

## 对外能力与返回契约

## 依赖关系与消费者

## 开发与验证

## 模块边界

## DO NOT

## 已知陷阱与历史教训
```

其中应包含：

- 模块用途及明确的非职责范围。
- 相关 README、架构、接口、数据库和运行手册路径。
- 开发、格式化、检查、测试和构建命令。
- 上下游依赖及允许的依赖方向。
- 数据库、schema、migration 或外部资源所有权。
- Docker 镜像、部署单元和健康检查信息（如适用）。
- 模块特有的风险、禁止事项以及从历史故障中得到的约束。

### 对外能力与返回契约

`packages/*` 及其他被复用模块必须记录其公开能力，不能只写“提供公共工具”。每项公开能力至少说明：

- 导出入口、函数、类型、组件、命令或协议名称。
- 输入参数及关键前置条件。
- 返回值、错误、事件或副作用。
- 同步/异步、可空性、幂等性和异常语义。
- 稳定性：internal、experimental、stable 或 deprecated。
- 对应源码、schema 或正式接口文档路径；`AGENTS.md` 只做摘要和索引，不复制整份 API 文档。

推荐使用表格：

```md
| 能力/导出 | 输入 | 返回/副作用 | 稳定性 | 契约来源 |
|---|---|---|---|---|
| `reserveQuota()` | 用户、请求与额度 | reservation 或结构化错误；预留额度 | stable | `src/...` |
```

文档中的返回契约必须可从源码、类型、schema 或测试验证。返回结构变化时，应同步更新提供方和所有消费者的文档与测试。

### 依赖关系与消费者

每个模块必须同时记录：

- **依赖谁**：直接依赖的 package、service、外部 API、数据库或基础设施，以及用途。
- **被谁依赖**：具体 app/service/package 及其具体功能，不得只写模糊的“多个服务使用”。
- **如何依赖**：源码 import、生成客户端、HTTP、事件、数据库或构建时工具。
- **契约边界**：版本、入口、协议、鉴权、超时、重试、幂等和错误处理要求。
- **变更影响**：公开能力变化时必须验证哪些消费者和功能。

推荐使用以下格式；其中名称仅用于演示字段，不表示对应模块已经存在：

```md
| 方向 | 模块 | 使用功能 | 依赖方式 | 契约/入口 | 变更后验证 |
|---|---|---|---|---|---|
| 被依赖 | `<consumer>` | `<具体消费者功能>` | workspace import | `<public entry>` | `<consumer checks>` |
| 依赖 | `<provider>` | `<调用能力>` | internal HTTP | `<versioned endpoint>` | `<contract + integration checks>` |
```

依赖记录只列直接且有实际证据的关系。Agent 不得根据目录名猜测消费者；应通过 import、调用点、workspace manifest、路由、生成配置或测试验证。依赖新增、删除、重命名或用途变化时，提供方与消费方的 `AGENTS.md` 必须双向更新。

### DO NOT

每个模块 `AGENTS.md` 必须有清晰、可执行的 `DO NOT`，记录该作用域内禁止的操作，例如：

- 禁止跨越服务或数据所有权边界。
- 禁止从私有源码路径导入，只允许使用公开入口。
- 禁止绕过封装、验证、鉴权、幂等或事务。
- 禁止使用已废弃 API、旧路径、旧容器或错误构建上下文。
- 禁止修改生成文件、共享 schema 或公共接口而不验证消费者。

`DO NOT` 应说明原因或指向对应历史教训，避免只有口号而没有适用条件。

### 已知陷阱与历史教训

踩坑记录应沉淀为可预防复发的知识，每项至少包含：

- 症状或失败结果。
- 根因。
- 正确做法。
- 禁止做法。
- 可验证的文件、测试、命令、issue 或事故记录。
- 仍然适用的条件；问题彻底消失后应删除或归档，不得永久保留失效警告。

推荐格式：

```md
### <简短标题>

- 症状：
- 根因：
- DO：
- DO NOT：
- 验证：
- 适用范围：
```

不要把聊天过程、未经验证的猜测或一次性的调试日志直接堆入 `AGENTS.md`。详细复盘放到正式文档，模块 `AGENTS.md` 只保留能够指导后续 Agent 的结论和索引。

维护要求：

- 创建新模块时，必须同时创建该模块的 `AGENTS.md`；缺少时模块骨架不算完成。
- 移动或重命名模块时，必须同步更新模块及上层 `AGENTS.md` 的索引和路径。
- 公开能力、返回契约、消费者、命令、服务边界、数据所有权或部署方式变化时，必须同步更新提供方与消费方的 `AGENTS.md`。
- 修复非显然的跨模块故障或重复踩坑后，必须把可复用结论记录到最近作用域的 `AGENTS.md`；详细过程放到正式复盘文档。
- 删除模块时，必须清理所有指向该模块 `AGENTS.md` 的索引，并从依赖方和消费者文档中移除对应关系。
- Agent 进入模块工作前，应从仓库根目录到目标目录逐层读取适用的 `AGENTS.md`。
- `AGENTS.md` 应保持简洁并以索引为主；详细说明放在模块自身文档或适当的 `.agents/` 文档中。

在 Monorepo 目录尚未获准创建时，不得为了满足本规范提前创建空模块或占位 `AGENTS.md`。本规范在模块实际创建或迁入时执行。

## 7. 共享代码规则

`packages/` 只用于具备明确跨项目价值和稳定边界的内容，例如：

- API schema、事件 schema 或生成的协议类型。
- 通用错误模型、日志字段及遥测约定。
- 与业务无关的基础工具和统一配置。
- 经确认需要复用的客户端 SDK。

禁止：

- 为消除少量重复而过早抽象。
- 把某个服务的业务逻辑放入 `packages/` 供其他服务直接调用。
- 共享 ORM client、数据库 repository 或数据库实体，以此绕过服务边界。
- 创建无边界的 `common`、`shared`、`utils` 大杂烩。
- 让 Go Executor 依赖仅适用于 Node.js 的内部实现。

新增共享包前必须说明：使用者、稳定 API、所有权、版本影响以及为何不能保留在单一服务内。

## 8. 依赖方向

允许的依赖方向：

```text
apps/services -> packages
apps/services -> versioned service contracts
infra/tools   -> workspace metadata
```

禁止的依赖方式：

- `packages` 反向依赖具体 app 或 service。
- 一个 service 导入另一个 service 的私有源码。
- 循环依赖。
- 前端直接依赖后端内部数据库模型。
- Executor 直接连接 Auth、Business 或 Logs 数据库。
- 通过共享数据库表替代正式服务接口。

服务间调用应通过内部 HTTP、消息或后续明确选定的协议，并保留鉴权、超时、重试、幂等和错误语义。

## 9. 多语言规则

TokenMP v3 是多语言 Monorepo：

- Executor 保持 Go。
- 其他服务或应用的语言与框架根据现有代码及后续决策确定。
- Node.js workspace 工具不得接管 Go 模块边界。
- Go workspace 使用 `go.work`；首个 Go module `services/auth` 已创建，模块路径 `github.com/tokenmp/v3/services/auth`，Go 1.26.5。后续 Go 服务通过独立变更新增 `go.work` 条目与模块级 `AGENTS.md`。
- 跨语言共享优先使用 OpenAPI、JSON Schema、Protobuf 等语言中立契约及代码生成，不复制手写类型作为唯一事实来源。

## 10. Workspace 与构建工具选型

已采用：

- Node.js 包管理器：pnpm 11.15.0。
- Workspace：`pnpm-workspace.yaml`。
- 任务编排：Turborepo 2.10.5。
- Node.js：26.4.0，通过 `mise.toml` 固定。
- TypeScript：6.0.3。
- Go：1.26.5，通过 `mise.toml` 固定。
- Go workspace：`go.work` 已创建，`use ./services/auth`。
- 当前模块数量：3（`packages/ui-tokens`、`packages/contracts`、`services/auth`）；其他顶层分区及其 `AGENTS.md` 仍属于仓库骨架，不代表具体应用、infra 或 tool 已实施。
- `services/auth/` 是首个 Go 服务，已实现 Auth Identity Flows（注册/登录/JWT/Refresh Token 轮换等，见 ADR 0005）。

详细理由见 `docs/adr/0001-monorepo-tooling.md` 与 `docs/adr/0004-auth-service-foundation.md`。

以下内容仍未决定，Agent 不得自行默认：

- 各 Go 服务的版本策略：统一版本或独立版本。
- 发布工具和远端缓存方案。
- 各应用与服务的运行时框架（首个 Go 服务的运行时框架已确认为 Chi + GORM，其他服务仍逐个确认）。
- Auth Go server generation 已固定使用 oapi-codegen v2.8.0（见 `packages/contracts/` 与 ADR 0006）；其他服务或语言的代码生成策略仍未决定。

Agent 提议未决选型时必须提供：

1. 当前代码和语言构成。
2. 候选方案及优缺点。
3. 对本地开发、CI、Docker 构建和部署的影响。
4. 锁文件、缓存、升级和维护成本。
5. 推荐方案及不采用其他方案的理由。
6. 可回滚的渐进实施步骤。

未经用户确认，不得把建议写成已落地事实或创建对应配置。

## 11. Docker 与部署边界

- Monorepo 中每个服务仍使用独立镜像。
- v3 Compose project 固定为 `tokenmp-v3`。
- 所有 v3 镜像、容器、网络和命名卷使用 `tokenmp-v3` 标识。
- 禁止制作包含所有服务的单一超级镜像。
- Docker build context、`.dockerignore` 和分层缓存策略应在实施方案中统一设计。
- 不得因为使用 Monorepo 而把全部源码无条件复制进每个镜像。
- Docker 通用规则以 `.agents/docker.md` 为准。
- 私有部署目标、旧环境路径和实时运行状态从可选 `.agents/local.md` 读取；没有本地上下文时必须先确认。

## 12. 配置与密钥

- 仓库只保存配置模板、schema 和非敏感默认值。
- 密钥、令牌、密码和生产配置不得提交。
- 每个可部署单元显式声明自己的环境变量。
- 环境变量命名和验证方式应统一，但服务只能读取自身所需配置。
- 本地、测试、预览和生产配置必须可区分，不得隐式共用。

## 13. 数据所有权

- 数据库属于服务边界，不属于 Monorepo 全局共享层。
- 每个 schema、migration 和 seed 必须有明确所有者。
- 服务不得直接运行其他服务的 migration。
- 跨库数据同步通过接口、事件或经确认的数据流程完成。
- 数据库整合、拆分或迁移必须单独设计备份、兼容和回滚方案。

## 14. 迁移原则

Monorepo 迁移应渐进执行，而不是一次性重写：

1. 盘点现有仓库、服务、语言、依赖、构建和部署入口。
2. 记录当前服务及数据边界，识别冲突和历史遗留。
3. 确认目标目录和工具选型，形成 ADR。
4. 在新的独立工作分支建立最小 workspace 骨架。
5. 每次只迁移一个边界明确的应用或服务。
6. 保持行为兼容，迁移期间避免同时重写业务逻辑。
7. 每一步验证开发、测试、构建、镜像和部署流程。
8. 确认新路径稳定后再清理旧结构。

结构迁移和业务重构必须拆分为不同任务、分支和提交。

## 15. Agent 工作规则

接到涉及 Monorepo、目录、workspace、依赖或构建的任务时，Agent 必须：

1. 先读取 `AGENTS.md`、`.agents/monorepo.md`、`.agents/gitflow.md` 和相关规则。
2. 检查当前分支和实施准入条件；未满足准入条件时不得实施结构迁移。
3. 先盘点现状，不根据历史路径假设当前代码存在。
4. 区分“已确定”“建议”“待确认”和“已实施”，不得混写。
5. 在修改结构前给出目录方案、依赖图、工具选型和迁移步骤。
6. 获得用户明确批准后，从最新 `main` 创建专用实施分支。
7. 实施时保持小步提交，不混入业务功能重写。
8. 修改共享配置后验证所有受影响服务，而不是只验证一个包。
9. 不自动安装全局工具，不修改用户级环境配置。
10. 未经明确要求，不提交、推送、合并或部署。

如果用户仅要求“规划”“定义”或“配置规则”，输出应停留在文档和方案层面，不得生成框架。

## 16. 新增模块前必须确认的事项

pnpm workspace、Turborepo 和基础 TypeScript 工具链已落地。Go workspace 已创建（`use ./services/auth`）。新增或迁移具体模块前，仍需按任务确认：

- 本次实际纳入仓库的应用、服务或 package 清单。
- 模块目录、职责、消费者和依赖方向。
- 运行时框架及公开契约。
- 如果涉及 Go，在 `go.work` 新增 `use` 条目并补充模块级 `AGENTS.md`。
- Auth Go server generation 已固定使用 oapi-codegen v2.8.0（见 `packages/contracts/` 与 ADR 0006）；其他服务或语言的代码生成策略仍未决定。
- 模块测试、CI、缓存和发布要求。
- Docker build context、镜像和部署边界。
- 现有代码来源、迁移顺序和兼容要求。

未完成对应模块的确认时，Agent 只能调研和提出方案，不得提前创建空模块或虚假契约。
